Question we tested

Baseline: loopback

  • Same split-inference fusion application.
  • Front and back communicate over 127.0.0.1.
  • No OAI radio/core path is involved.
  • Purpose: isolate application compute and UDP packaging overhead.

Test path: OAI 5G

  • Same application and model checkpoint.
  • Front clients bind to OAI UE tunnel IPs.
  • Back half runs in the OAI receiver container.
  • Purpose: measure added transport delay and result loss over the OAI path.

The goal is not only to show OAI is slower, but to locate whether the extra latency is compute, payload encoding, backend inference, or network/queue behavior.

OAI network configuration used

5G/RAN setup

  • OAI RAN in RF simulator mode: --rfsim.
  • gNB config: gnb.sa.band78.fr1.106PRB.usrpb210.conf.
  • Band: 78; numerology: 1.
  • PRBs: 106, corresponding to the 20 MHz FR1 profile used in this setup.
  • DL carrier frequency argument: 3619200000.

Multi-UE transport mapping

  • UE1: oaitun_ue1 / 10.0.0.2.
  • UE2: oaitun_ue2 / 10.0.0.3.
  • Receiver container: oai-perception-rx / 192.168.70.140.
  • Core network examples use 192.168.70.x; ext-DN is 192.168.70.135.
  • Fusion feature transport: UDP, 6000-byte chunks, zlib, per_channel_uint8.

Where the latency lives

Backend inference is small and stable. The OAI gap is dominated by the transport/queue component estimated as RTT - back_ms.

Interpretation

What the evidence supports

  • OAI is about 5x to 6x higher average RTT than loopback.
  • OAI p95 RTT reaches roughly 450-538 ms.
  • Loopback receives every result; OAI receives only about 69-82%.
  • Single-UE OAI is still high latency, so the second stream is not the only cause.

What it does not prove yet

  • It does not identify the exact radio-layer bottleneck.
  • It does not separate OAI tunnel queueing from UDP reassembly behavior.
  • It does not prove the 20 MHz / 106 PRB profile is saturated in every run.
  • It does not yet use T-tracer timing to localize MAC/RLC/PDCP delays.

Network sampler evidence

Dual OAI captured traffic

  • UE1 avg TX: 8.34 Mbps; p95 TX: 27.86 Mbps; max TX: 37.13 Mbps.
  • UE2 avg TX: 10.03 Mbps; p95 TX: 28.34 Mbps; max TX: 37.84 Mbps.
  • UE tunnel TX drops were observed: 6023 on UE1 and 10272 on UE2.
  • Ping RTT was much lower than application RTT: about 19-21 ms average.

Caution on solo sampler runs

  • The solo OAI app logs clearly received fusion results.
  • However, the solo network sampler summaries look nearly idle.
  • So the solo network counters should not be used for final claims until the sampler/interface timing is checked.
  • The app-level comparison remains valid because it is logged by the fusion clients themselves.

Why loopback is still useful

  • Loopback is not a deployment claim. It is the controlled baseline.
  • It exercises the same model split, feature serialization, UDP chunking, result decoding, and visualization path without the OAI network.
  • Because loopback RTT is low and stable, the fusion application and back-half inference are not the main reason OAI playback feels laggy.
  • The difference between loopback and OAI gives a clean estimate of the transport-side penalty that needs deeper tracing.

The story for the supervisor: the logging pipeline is working, the visual lag is measurable, and the current evidence points toward network/queue behavior rather than model compute.

Next steps

Root-cause tracing

  • Repeat dual OAI with T-tracer enabled on gNB and UE.
  • Join app frames to radio/tunnel events using the same run group.
  • Check where delay builds: UE tunnel, PDCP/RLC/MAC, container path, or UDP reassembly.
  • Fix solo network sampler capture so counters match app traffic.

System experiments

  • Try wider/UL-heavy RAN configuration after baseline tracing.
  • Run controlled 20 MHz vs wider-bandwidth comparisons if the config is stable.
  • Evaluate lower payload settings and non-blocking display/result handling.
  • Move toward parked-ego fusion experiments once transport behavior is characterized.
