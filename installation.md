> **Purpose:** end-to-end installation reference for V2Xverse + CoDriving on
> a clean machine. Includes sanity checks at every step, troubleshooting,
> and the specific gotchas we expect on Blackwell-class GPUs.

## 0. Critical preconditions / heads-up

Read these *before* starting. They have bitten us elsewhere.

### 0.1. CARLA version isolation

V2Xverse needs **CARLA 0.9.10.1** — *not* CARLA 0.10, which our team uses.
This is FINE: V2Xverse's `setup_carla.sh` downloads 0.9.10.1 into a *local*
folder inside the repo. **Do not** install it system-wide or `easy_install`
its egg into the team's CARLA 0.10 venv.

### 0.2. PyTorch 1.10.1 vs Blackwell GPU (RTX 5090, sm_120)

V2Xverse pins to PyTorch 1.10.1 + CUDA 11.3. That version of PyTorch was
released *years before* Blackwell, so it has **no compiled kernels for
sm_120**. Symptoms when running on an RTX 5090:

```
RuntimeError: CUDA error: no kernel image is available for execution on the device
```

This is a **runtime** issue, not an install issue. The install will succeed.
After install we run a 30-second GPU sanity check (§3.1). If it fails, we
have three escape routes:

1. **Run on a different machine** with a pre-Blackwell GPU (RTX 30/40-series).
   Supervisor offered access to his machine for exactly this kind of issue.
2. **CPU-only mode** (`CUDA_VISIBLE_DEVICES=""`) for install verification only.
   Slow but functional.
3. **Patch to a newer PyTorch** (≥2.4). High risk — may break spconv 1.2.1
   and other pinned deps.

### 0.3. Conda environment isolation

V2Xverse goes in a **dedicated conda env** (`v2xverse`). Everything stays in
`~/miniconda3/envs/v2xverse/`. Will not touch:
- Our CARLA 0.10 venv (`carla_0_10_venv`)
- System CUDA 13.x
- Other team members' work

### 0.4. Do not run their 15-process dataset generation

Their dataset can be generated by spawning **15 parallel CARLA processes**.
That is heavy and we don't need it — they provide a pre-generated dataset
on HuggingFace. Use the download path (§7), not the generate path.

### 3.1. Install PyTorch + CUDA runtime — and IMMEDIATELY sanity-check the GPU

> **2026-05-21 update — original PyTorch 1.10.1 pin replaced.**
> On a Blackwell GPU (RTX 5090, sm_120), PyTorch 1.10.1 hangs irrecoverably
> (see §12.2). We patched to PyTorch 2.x nightly with CUDA 12.x. This is a
> documented deviation from V2Xverse's README; the patched version is what
> the rest of this doc now assumes.

**Use Python 3.10 in step 3.0 (not 3.7)** — PyTorch 2.x dropped Python 3.7.

**Install command (Blackwell-compatible):**
```bash
# Stable PyTorch with CUDA 12.8 (may not have Blackwell support depending on date):
pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu128

# If stable lacks sm_120 still, use nightly (confirmed working 2026-05-21):
pip install --pre torch torchvision torchaudio \
    --index-url https://download.pytorch.org/whl/nightly/cu128
```

**Confirmed working as of 2026-05-21:** `torch 2.9.0.dev20250813+cu129` on RTX 5090.

**Original V2Xverse install (kept here for reference / non-Blackwell machines):**
```bash
# Only use this on pre-Blackwell GPUs (RTX 30-series or older):
conda install pytorch==1.10.1 torchvision==0.11.2 torchaudio==0.10.1 \
    cudatoolkit=11.3 -c pytorch -c conda-forge -y
```

**Sanity check — run this right after install (takes 5 seconds):**
```bash
python - <<'PY'
import torch
print("torch version:", torch.__version__)
print("CUDA available:", torch.cuda.is_available())
if torch.cuda.is_available():
    print("GPU:", torch.cuda.get_device_name(0))
    print("Compute capability:", torch.cuda.get_device_capability(0))
    try:
        x = torch.zeros(10, device='cuda') + 1
        print("Tiny GPU op succeeded:", x.cpu().tolist())
    except Exception as e:
        print("GPU op FAILED:", type(e).__name__, str(e)[:200])
PY
```

**Possible outcomes:**

| Output | Diagnosis | Action |
|---|---|---|
| `Tiny GPU op succeeded: [1, 1, 1, ...]` | All good | Continue with §4 |
| `GPU op FAILED: RuntimeError CUDA error: no kernel image is available for execution on the device` | Blackwell sm_120 problem — predicted | **STOP. Go to §10 (Escape Routes).** |
| `CUDA available: False` | Driver or library issue | Check `nvidia-smi`; check `LD_LIBRARY_PATH` |
| `RuntimeError: CUDA driver version is insufficient...` | Driver too old | Update NVIDIA driver (root needed) |

### 3.2. Why this works without touching system CUDA

Conda's `cudatoolkit=11.3` package installs CUDA 11.3 *runtime libraries*
into the env folder. The bundled libraries are what PyTorch dynamically
loads. The system's `/usr/local/cuda-13.x/` is irrelevant — different
versions, different paths, no conflict.

## 5. Install spconv 2.x (replaces V2Xverse's pinned 1.2.1)

> **2026-05-21 update — V2Xverse's pinned `spconv 1.2.1` does not work with PyTorch 2.x.**
> We install `spconv 2.x` (modern API) instead. V2Xverse's `opencood` code will
> need a patch for the changed import path — that's handled in §6.

**Install (try newest CUDA 12 wheel first):**
```bash
conda activate v2verse
pip install spconv-cu121
# If that fails, fall back to cu120:
# pip install spconv-cu120
```

**Sanity check (covers import + a small GPU sparse-conv forward pass):**
```bash
python - <<'PY'
import torch
import spconv.pytorch as spconv
print("spconv import OK")

features = torch.randn(10, 4, device='cuda')
indices  = torch.randint(0, 32, (10, 4), device='cuda', dtype=torch.int32)
indices[:, 0] = 0
sp_tensor = spconv.SparseConvTensor(features, indices, spatial_shape=[32, 32, 32], batch_size=1)
print("SparseConvTensor created on GPU")

layer = spconv.SubMConv3d(4, 8, kernel_size=3, padding=1).cuda()
out = layer(sp_tensor)
print("Sparse conv forward succeeded; output feature shape:", out.features.shape)
PY
```

**API change vs V2Xverse's expected version:**
| V2Xverse (spconv 1.x) | spconv 2.x replacement |
|---|---|
| `import spconv` | `import spconv.pytorch as spconv` |
| `spconv.SparseConvTensor(...)` | Same name, same args (compatible) |
| `spconv.SubMConv3d`, `SparseConv3d` | Same names under `spconv.pytorch` |
| `spconv.SparseSequential` | Same |
| Several internal helpers | May differ — patch on a case-by-case basis |

We will rewrite V2Xverse's `import spconv` lines in §6 once we know which files need it.

**Original spconv 1.2.1 install (kept for reference / non-Blackwell machines):**
```bash

pip install spconv==1.2.1
# (May need to build from source — see prior version of this doc.)
```

## 12. Known issues log

Update this section as we hit and resolve issues.

### 12.1. `ImportError: undefined symbol: iJIT_NotifyEvent` on `import torch`
**Cause:** PyTorch 1.10.1 was compiled against an older MKL that had `iJIT_NotifyEvent`
in `libittnotify`. MKL 2025+ removed that symbol. Conda may pull a too-new MKL when
resolving the env.

**Fix (confirmed working 2026-05-21):**
```bash
conda activate v2verse           # (or your env name)
conda install -c conda-forge "mkl<2025" -y
```
If conda resolver fights you, fall back to:
```bash
conda install -c conda-forge mkl=2024.0.0 -y
# or
pip install "mkl<2025"
```
Then re-import torch. Issue should disappear.

### 12.3. CARLA Python API: no Python 3.10 wheel for 0.9.10.1 — **CONFIRMED BLOCKER on this box**

**Symptom:** V2Xverse pins CARLA 0.9.10.1. The simulator binary downloads and
starts fine. But `import carla` in Python 3.10 fails because:
- `pip install carla==0.9.10` → no matching distribution (PyPI has 0.9.5, 0.9.15, 0.9.16 only on Python 3.10)
- The `.egg` files shipped with CARLA 0.9.10.1 are for Python 2.7 and 3.7 — incompatible ABI with Python 3.10

**Workarounds tested:**
- `pip install carla==0.9.16` → installs cleanly. Connects to 0.9.10.1 server. `get_*_version()` works. But `client.get_world()` fails with `RuntimeError: rpc::rpc_error during call in function get_sensor_token` — newer client calls RPC functions that the 0.9.10.1 server doesn't implement.
- `pip install carla==0.9.15` → same failure mode, same RPC mismatch.
- `pip install carla==0.9.5` → too old, expected to fail before 0.9.10.

**Conclusion:** there is no Python 3.10 client that talks to the 0.9.10.1 server.

**Possible resolutions (in order of effort, lowest first):**
1. **Migrate to supervisor's machine** — fastest if available. His environment
   already matches V2Xverse's stated stack.
2. **Upgrade simulator to 0.9.16** to match the pip-installed client. Re-download
   CARLA. Then V2Xverse code likely needs further patches for scenarios /
   blueprints that changed between 0.9.10 and 0.9.16.
3. **Build the CARLA 0.9.10.1 Python API from source for Python 3.10.**
   Multi-hour effort. Needs CARLA source tree, Boost, libpng, etc. May hit
   Blackwell-related compiler issues on top.

**Decision logged 2026-05-21:** escalated to supervisor for direction. Each
patch is doable but cumulative divergence from V2Xverse's environment is
high enough that reproducing their published results becomes uncertain.

### 12.2. Blackwell GPU (RTX 5090, sm_120) hangs PyTorch 1.10.1 — **PREDICTED, CONFIRMED**
**Symptom:** `import torch` works; `torch.cuda.is_available()` returns True; but
any actual GPU operation (`torch.zeros(10, device='cuda') + 1`) **hangs the
Python process indefinitely**. PyTorch prints the warning:
> *NVIDIA GeForce RTX 5090 with CUDA capability sm_120 is not compatible with the
> current PyTorch installation. The current PyTorch install supports CUDA
> capabilities sm_37 sm_50 sm_60 sm_61 sm_70 sm_75 sm_80 sm_86.*

**Cause:** PyTorch 1.10.1 (Oct 2021) was compiled with `TORCH_CUDA_ARCH_LIST`
that only includes up to sm_86 (Ampere / RTX 30-series). Blackwell (sm_120, RTX
5090) didn't exist when 1.10.1 was built, so there are no precompiled CUDA
kernels for it. The driver hangs trying to dispatch.

**Recovery from hung process** (Ctrl+C does not work, the syscall is uninterruptible):
```bash
# In another terminal:
ps aux | grep "python" | grep -v grep
kill -9 <PID>
# Or, more aggressively:
pkill -9 -f python
```
`Ctrl+\` (SIGQUIT) is sometimes effective in the original terminal.

**Resolution paths** (see §10 Escape Routes):
- **Recommended:** run V2Xverse on a machine with a pre-Blackwell GPU. Supervisor
  offered access for exactly this kind of issue.
- **CPU-only on this box** (`export CUDA_VISIBLE_DEVICES=""`) — fine for install
  verification and reading model architecture, too slow for real workloads.
- **Patch V2Xverse to PyTorch ≥2.4** — high risk; breaks spconv 1.2.1 and others.

This is a *fundamental* PyTorch version constraint, not a settings issue. No
workaround inside PyTorch 1.10.1.
